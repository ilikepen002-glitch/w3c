# 启动时序导致阴影与农民未初始化

## 问题现象与影响范围

- 进入地图后仍然有战争迷雾/黑色阴影。
- 玩家开局没有刷出 TD 农民，因此无法建塔。
- 两个症状同时出现，看起来像“启动脚本根本没跑”，但编译、测试和部署脚本都能成功。

影响范围包括 `TDInit` 地图启动流程、`Hello` 入口、玩家路线激活检测，以及所有依赖 `tdStartGame()` 的开局初始化行为。

## 关键环境和触发条件

- 项目 `scriptMode: JASS`
- `wurst.build` 目标补丁为 `v1.28`
- 本地测试部署目录对应的是 Warcraft III `1.27a`
- 启动入口通过 `Hello.wurst` 中的 `doAfter(0.00)` 间接调用 `tdStartGame()`
- `tdCreatePads()` 只有在 `tdDetectActiveLanes()` 检测到 `PLAYER_SLOT_STATE_PLAYING` 后才会给对应玩家刷农民

## 曾尝试但无效的方法，以及它们为什么无效

- 直接怀疑农民 rawcode 或出生点坐标：
  - 这解释不了“阴影也没关”这个同时出现的症状。
- 直接改成读取 Start Location 刷原生 `'hpea'`：
  - 不符合项目当前架构。项目实际使用自定义农民 `uF01`，出生点也来自 `gg_rct_td_farmer_p1..p4`。
- 只看源码是否有 `FogEnableOff()` 和 `createUnit()`：
  - 编译产物里这些调用都存在，但不能证明它们在实机上触发时机正确。

## 最终根因

- `tdStartGame()` 的运行时机过早，过度依赖 `doAfter(0.00)` 在不同补丁/运行环境下的稳定性。
- 在较旧实机环境中，0 秒延迟触发时，玩家槽位状态可能还没有稳定为 `PLAYER_SLOT_STATE_PLAYING`。
- 结果是：
  - `tdDetectActiveLanes()` 可能暂时检测不到有效玩家；
  - 农民不会生成；
  - 去阴影逻辑也可能发生在过早时机，表现为进图后仍有黑色阴影。

## 最终解决方案及关键改动

- `TDInit.wurst`
  - 将 `tdMapBootstrapDelay()` 从 `0.0` 改为 `0.10`
  - 新增：
    - `tdMapBootstrapRetryInterval()`
    - `tdMapBootstrapMaxRetries()`
    - `tdFinalizeGameplayBootstrap()`
    - `tdBootstrapGameplayWhenPlayersReady(int retryCount)`
  - 启动流程改为：
    - 先短暂等待；
    - 检测是否已经有激活路线；
    - 若没有，则按固定间隔重试；
    - 一旦检测到玩家路线，或达到最大重试次数，再统一执行去阴影、镜头和玩法初始化
  - 新增 `tdGameplayStartupCompleted`，避免重复初始化
- `TDInitTests.wurst`
  - 新增启动延迟与重试策略测试，确保后续不会再无意回退到 0 秒启动

## 验证命令、验证结果和残余风险

- `tools\Run-WurstChecks.bat test TDInitTests`
  - 结果：通过
- `tools\Run-WurstChecks.bat`
  - 结果：通过
- `build-and-deploy-test-map.bat`
  - 结果：构建并部署成功

残余风险：

- 当前根因判断来自“编译产物已包含正确逻辑”与“实机症状表现”的交叉分析，而不是直接在 Warcraft 进程里打断点。
- 如果部署后的最新地图仍然复现，则下一步应在开局加入更显眼的运行时诊断文本，直接确认：
  - 是否执行到了 `tdStartGame()`
  - 首次检测到的 `tdCountActiveLanes()` 是多少
  - 是否触发了重试兜底

## 可复用经验与可提炼技能

- 当多个开局系统同时失效时，优先怀疑启动时序，而不是分别修农民、镜头、阴影等表面症状。
- “编译产物里有代码”不等于“实机在正确时机执行了代码”；需要把入口时序也纳入排查。
- 对依赖 `GetPlayerSlotState()` 的启动逻辑，应避免 0 秒延迟直冲，优先使用“短延迟 + 条件重试”。

可提炼技能：

- Warcraft III/Wurst 启动时序排查
- `PLAYER_SLOT_STATE_PLAYING` 依赖的稳健初始化策略
- “编译通过但实机像没跑脚本”的差异化诊断流程
