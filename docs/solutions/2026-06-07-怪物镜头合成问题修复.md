# 怪物移动、血条、镜头和合成问题修复

## 问题现象与影响范围

- 怪物受击或巡路时表现为走走停停，路线推进不稳定。
- 怪物血条显示不可靠，需要保留 Warcraft 原生血条，而不是依赖自定义 texttag。
- 开局镜头距离容易被引擎初始化覆盖，且滚轮拉高需要限制在 3 个额外高度内。
- 工人放入 2 个或 3 个装备时，合成判定和玩家理解的前 3 个物品槽不一致。

影响范围包括 `TDSpawn` 路线命令、`TDWaves` 怪物物编生成、`TDInit` 开局镜头、`TDInventoryRuntime` 合成槽位读取，以及对应测试。

## 关键环境和触发条件

- 当前项目 `scriptMode: JASS`，构建和部署必须通过 `build-and-deploy-test-map.bat`。
- 怪物由 `TDWaves` compiletime 生成，并由 `TDSpawn` 周期性检查路点。
- Warcraft III 会在地图载入后进行默认镜头重置，过早设置的镜头字段可能被覆盖。
- 工人库存是 6 格物品栏，合成需求按源码文案中的“前 3 格”理解。

## 曾尝试但无效的方法

- 单纯关注怪物是否有攻击能力不够完整。对象生成里已经关闭了攻击、主动攻击范围和逃跑，但路线 watcher 仍每 0.2 秒重复下达 move 命令，持续打断寻路状态。
- 将合成问题归因于空物品判空不符合当前源码。当前运行时代码在读取 `item.getTypeId()` 前已经做了 null 和装备判定。
- 只在地图启动时立即设置镜头不稳定，因为引擎默认镜头初始化可能随后覆盖设置。

## 最终根因

- 怪物卡顿的直接根因是 `tdAdvanceEnemyWaypoint` 每次巡检都会调用 `tdOrderEnemyAlongLane`，而旧实现无条件重新发布 `move`。即使怪物没有攻击能力，也会被频繁重置当前移动命令。
- 血条风险来自对象字段没有显式锁定：需要避免继承 `Aloc` 等普通技能，并确保选择缩放为正值。
- 镜头缺少启动后的延迟重拉和周期性距离上限夹取。
- 合成读取第三个输入时映射到了 slot 3，而玩家和文案中的“前 3 格”应对应 slot 0、1、2。

## 最终解决方案及关键改动

- `TDSpawn` 新增 `tdEnemyShouldIssueLaneMove`，只有当前命令不是 move 或刚刚抵达路点并切换目标时才重新发布移动命令。
- `TDWaves` 新增怪物对象生成配置函数，显式设置：
  - `attacksEnabled = 0`
  - `acquisitionRange = 0`
  - `collisionSize = 12`
  - `selectionScale = 1.0`
  - `normalAbilities = ""`
- `TDInit` 新增 0.10 秒镜头重拉，并每 0.15 秒本地夹取 `CAMERA_FIELD_TARGET_DISTANCE` 到 3200 到 4400。
- `TDInventoryRuntime` 将合成可见输入槽改为 0、1、2。
- 对应补充 `TDSpawnTests`、`TDWavesTests`、`TDInitTests`、`TDCraftingTests`。

## 验证命令、验证结果和残余风险

- `tools\Run-WurstChecks.bat`
  - 结果：类型检查通过，全部测试通过。
- `.\build-and-deploy-test-map.bat`
  - 结果：地图构建成功，生成 `_build\MyFirstWurstMap.w3x.w3x`。
  - 残余风险：部署复制到 `Maps\Test\深渊守望.w3x` 时目标文件被 `War3` 进程占用，脚本输出 Copy-Item 错误但返回 0。需要关闭 Warcraft III 后再次运行脚本完成部署。

## 可复用经验与可提炼技能

- TD 怪物走位问题要同时查对象 AI 字段和触发器下令频率；禁攻击不是全部答案。
- 血条问题优先查 `Aloc`、选择缩放、模型 bounds；在代码生成物编时应显式写入关键字段，避免继承父单位的隐藏状态。
- 开局镜头设置至少需要一次游戏开始后的延迟重拉；滚轮高度限制应使用本地 camera field 读取和夹取，避免同步状态依赖本地镜头值。
- 物品合成问题先区分“空槽判空”和“槽位映射错误”，不要把所有合成失败都归因于 null。
- 可提炼技能：Warcraft TD 运行时问题排查、Wurst compiletime 物编安全字段检查、Warcraft 镜头初始化与滚轮限制模式。
